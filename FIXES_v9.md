## ✅ ما تم إصلاحه في الكود

### 2) كلمة سر الموظف/الكاشير لا تتغيّر
- **السبب الجذري:** تأثير "التشفير عند أول تشغيل" في `store.js` كان يرفع المستخدمين الافتراضيين إلى Supabase على كل جهاز جديد، فيكتب فوق كلمات المرور المعدّلة بالافتراضية (سباق مع تأثير التحميل).
- **الحل:** أُزيل الرفع من ذلك التأثير نهائياً. الآن تُبذَر الافتراضيات إلى Supabase **فقط عندما يكون جدول `profiles` فارغاً تماماً** (داخل تأثير التحميل).

### 3) + 4) توجيه الطلبات بين المحطات (بار / أراكيل)
- **السبب:** حالة واحدة لكل الطلب، فأي محطة كانت تُنهي الطلب كاملاً.
- **الحل:** علم `prepared` لكل صنف. البار يُحضّر المشروبات فقط، الأراكيل تُحضّر النرجيلة فقط، والطلب يصبح "جاهزاً" كلياً **فقط عند اكتمال كل أصناف المحطات**. (لا حاجة لتعديل قاعدة البيانات — `items` من نوع JSONB.)
- ملفات: `App.jsx` (BarTab, HookahTab, OrdersTab) + `store.js` (مزامنة تغييرات الأصناف).

### 1) فشل المزامنة الصامت
- أخطاء Supabase لم تعد تُبتلع بصمت — تُرسَل عبر حدث `nc-sync-error`.

يتطلب تعديل هوية سطر السلة في شاشتَي الطلب — يُنفَّذ كخطوة تالية مع تحقّق بناء منفصل.

## الاختبار قبل الدمج
1. شغّل `npm install && npm run build` (يجب أن ينجح).
2. اختبر تغيير كلمة سر موظف ثم دخوله بها.
3. اطلب طلباً مختلطاً (مشروب + أركيلة): تأكّد أن البار يُنهي المشروب والأراكيل تُنهي النرجيلة، وأن الطلب لا يصبح "جاهزاً" إلا بعد الاثنين.
4. بعد تشغيل ملف SQL: تأكّد من ظهور طلب الزبون على جهاز البار.
